# WaterDem Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build WaterDem — a collaborative houseplant care app with AI plant identification, weather-aware watering countdowns, and read-only garden sharing — per the Leafly build brief (renamed WaterDem), on a newly created Supabase cloud project.

**Architecture:** Next.js 15 App Router with server route handlers for all privileged work (AI, DB, auth). Custom nickname/password auth (bcryptjs + jose JWT in an httpOnly cookie). The database is a new Supabase project (region `ap-southeast-1`) accessed via direct Postgres (postgres.js) so every request runs inside a transaction that sets `app.current_user_id`, making Row Level Security the primary enforcement layer (`FORCE ROW LEVEL SECURITY` since the server connects as table owner). AI identification/advice uses the OpenAI API (gpt-4o, vision) — provider isolated in one file (`src/lib/ai.ts`) for easy swap. Weather from open-meteo (no key). UI is bespoke glassmorphism: Tailwind v4 design tokens, Fraunces + Inter via next/font, Lucide icons only (no emoji), Framer Motion.

**Tech Stack:** Next.js 15.5 (App Router, TS), Tailwind CSS v4, Supabase (Postgres + Storage), postgres.js, bcryptjs, jose, OpenAI gpt-4o, open-meteo, lucide-react, framer-motion.

**Deviations from brief (user-approved):** App name is **WaterDem** (not Leafly). AI provider is **OpenAI gpt-4o** (not Anthropic) — user's choice; `OPENAI_API_KEY` placeholder left in `.env.local` for the user to fill.

## RLS design (applies supabase-postgres-best-practices)

- Identity: `set_config('app.current_user_id', <uuid>, true)` inside each transaction; policies read `(select nullif(current_setting('app.current_user_id', true), '')::uuid)` — scalar subquery so it evaluates once per query, not per row.
- `ENABLE` **and** `FORCE ROW LEVEL SECURITY` on all four tables (server connects as table owner `postgres`; without FORCE, owner bypasses RLS).
- Per-operation policies; indexes on every policy-referenced column: `plants(owner_id)`, `water_logs(plant_id)`, `garden_shares(owner_id, viewer_id)` (unique), `users(lower(nickname))` (unique).
- `users` SELECT is unrestricted at row level, but the app only ever selects `id, nickname, garden_is_public` for other users; `password_hash` never leaves `src/lib/db.ts` queries scoped to login. Column-level protection: `REVOKE`/grants are impractical for the owner connection, so enforcement is "no query selects it" + policies restrict UPDATE to own row.
- Visibility rule (MVP): viewers (invited or public) see exactly `is_public = true` plants; invited viewers work even when `garden_is_public = false`.

## Countdown / status math (`src/lib/countdown.ts`)

- `effectiveFreqDays = water_freq_days * wateringFactor(weather)` where factor = 0.8 if hot (>25°C) or dry (<40% RH), 0.7 if both, else 1.
- `elapsed = now - last_watered` (no `last_watered` → treat as due now); `remaining = effectiveFreqDays*24h - elapsed`.
- Status: `ok` (green) if remaining ratio > 1/3, `soon` (amber) if 0 < ratio ≤ 1/3, `overdue` (red) if ≤ 0. Colors only on the 1px bar + 6px dot at low-opacity fills.

**Migration SQL (complete):**

```sql
create table users (
  id uuid primary key default gen_random_uuid(),
  nickname text unique not null check (char_length(nickname) between 3 and 20),
  password_hash text not null,
  garden_is_public boolean not null default false,
  location_lat double precision,
  location_lon double precision,
  location_label text,
  created_at timestamptz not null default now()
);
create unique index users_nickname_lower_idx on users (lower(nickname));

create table plants (
  id uuid primary key default gen_random_uuid(),
  owner_id uuid not null references users(id) on delete cascade,
  name text not null,
  species text,
  common_name text,
  image_url text,
  icon_key text,
  water_freq_days int not null default 7 check (water_freq_days between 1 and 90),
  care_level text check (care_level in ('easy','moderate','expert')),
  light text check (light in ('low','medium','bright')),
  humidity text check (humidity in ('low','medium','high')),
  soil_check text,
  weather_note text,
  nutrients jsonb not null default '[]',
  weekly_tips jsonb not null default '[]',
  fun_facts jsonb not null default '[]',
  is_public boolean not null default false,
  last_watered timestamptz,
  created_at timestamptz not null default now()
);
create index plants_owner_id_idx on plants (owner_id);

create table water_logs (
  id uuid primary key default gen_random_uuid(),
  plant_id uuid not null references plants(id) on delete cascade,
  watered_at timestamptz not null default now()
);
create index water_logs_plant_id_idx on water_logs (plant_id, watered_at desc);

create table garden_shares (
  id uuid primary key default gen_random_uuid(),
  owner_id uuid not null references users(id) on delete cascade,
  viewer_id uuid not null references users(id) on delete cascade,
  created_at timestamptz not null default now(),
  unique (owner_id, viewer_id)
);
create index garden_shares_viewer_idx on garden_shares (viewer_id);

create or replace function current_app_user() returns uuid
language sql stable as $$
  select nullif(current_setting('app.current_user_id', true), '')::uuid
$$;

alter table users enable row level security;
alter table users force row level security;
alter table plants enable row level security;
alter table plants force row level security;
alter table water_logs enable row level security;
alter table water_logs force row level security;
alter table garden_shares enable row level security;
alter table garden_shares force row level security;

-- users: anyone can read public identity fields (app only selects id/nickname/garden_is_public);
-- signup INSERT allowed pre-auth; UPDATE only own row.
create policy users_select on users for select using (true);
create policy users_insert on users for insert with check (true);
create policy users_update on users for update
  using (id = (select current_app_user()))
  with check (id = (select current_app_user()));

-- plants
create policy plants_select on plants for select using (
  owner_id = (select current_app_user())
  or (
    is_public = true and (
      exists (select 1 from users u where u.id = plants.owner_id and u.garden_is_public)
      or exists (select 1 from garden_shares gs
                 where gs.owner_id = plants.owner_id
                   and gs.viewer_id = (select current_app_user()))
    )
  )
);
create policy plants_insert on plants for insert
  with check (owner_id = (select current_app_user()));
create policy plants_update on plants for update
  using (owner_id = (select current_app_user()))
  with check (owner_id = (select current_app_user()));
create policy plants_delete on plants for delete
  using (owner_id = (select current_app_user()));

-- water_logs: visible when parent plant is visible; writable by owner only
create policy water_logs_select on water_logs for select using (
  exists (select 1 from plants p where p.id = water_logs.plant_id)  -- delegates to plants RLS
);
create policy water_logs_insert on water_logs for insert with check (
  exists (select 1 from plants p where p.id = water_logs.plant_id
          and p.owner_id = (select current_app_user()))
);
create policy water_logs_delete on water_logs for delete using (
  exists (select 1 from plants p where p.id = water_logs.plant_id
          and p.owner_id = (select current_app_user()))
);

-- garden_shares: owner manages; viewer may see their own invite rows
create policy garden_shares_select on garden_shares for select using (
  owner_id = (select current_app_user()) or viewer_id = (select current_app_user())
);
create policy garden_shares_insert on garden_shares for insert
  with check (owner_id = (select current_app_user()));
create policy garden_shares_delete on garden_shares for delete
  using (owner_id = (select current_app_user()));
```

Note: `water_logs_select` policy's `exists (select 1 from plants ...)` runs under the caller's RLS context because the function/policy is not security-definer — plants RLS applies inside the subquery, so visibility is exactly "parent plant visible to me". (Owner connection is subject to RLS via FORCE.)

```ts
import postgres from "postgres";
const sql = postgres(process.env.DATABASE_URL!, { prepare: false, max: 5 });
export async function withUser<T>(userId: string | null, fn: (tx: postgres.TransactionSql) => Promise<T>): Promise<T> {
  return sql.begin(async (tx) => {
    await tx`select set_config('app.current_user_id', ${userId ?? ""}, true)`;
    return fn(tx);
  }) as Promise<T>;
}
export default sql;
```
